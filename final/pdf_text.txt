7. Risks & Challenges (≤ 5 bullets, 1 line each)  
• Pipeline stall timing: stall_read is a registered version of cop_busy (1 -cycle slip), meaning one 
extra instruction enters the pipeline before it freezes; must be handled carefully to avoid 
corrupting register state when the CPU dispatches to the coprocess or. 
• Signed vs unsigned INT8 arithmetic: all MAC lane signals must be declared signed [7:0] in 
Verilog; omitting the signed keyword causes Vivado to infer unsigned multiplications, producing 
silently wrong dot -product results for negative weights.  
• BRAM read latency: all BRAMs have 1 -cycle synchronous read latency; the coprocessor FSM 
must present the read address one state before consuming the data, otherwise the first data word 
per neuron will be garbage.  
• FPGA timing closure with wide bus: the 8 -lane 64 -bit vec_a / vec_b buses between the BRAM 
and MAC engine may fail timing at 100 MHz if routing is unfavourable; mitigation is to add a 
pipeline register stage between BRAM output and the DSP inputs.  
• Simulation vs FPGA mismatch: uninitialized BRAM contents and reset sequencing differences 
are common causes of passing simulation but failing FPGA execution; ILA probes on cop_state, 
mac_state, and acc will be set up for FPGA -side debugging.  
 
8. Response to TA feedback (for the project abstract)  
Point 1 — How do you handle pipeline stalls and CPU –coprocessor 
synchronization?  
In our project, the RISC -V CPU runs a C program (full_inference.c) that dispatches neural network layer 
computations to the coprocessor entirely through ordinary store -word (SW) instructions. The C code 
does: COP[5] = 128 (LAYER_ROWS), COP[6] = 784 (LAYER_ COLS), then COP[0] = 1 (START). The 
compiler compiles each of these into a SW instruction targeting addresses in the 0xC000_0000 range — 
there is no special ISA extension. The CPU has no idea it is talking to a coprocessor; it just sees a 
memory write.  
 
Because the hardware auto -stalls the CPU via cop_busy, the C program does not need to 
waste cycles polling a status register. The CPU simply goes to sleep when START is written 
and wakes up exactly when the result is ready to be read.  The stall freezes the CPU for exactly as 
long as the coprocessor needs, making the hardware acceleration completely transparent to the C 
program.  
 
Mechanically, when the START write reaches the WB stage of the pipeline, the MMIO address decoder in 
top_fpga.v detects the 0xC0 prefix on dmem_write_address and asserts is_cop_write. The 
coprocessor_top module latches start_reg  = 1 on the next rising edge. The master FSM immediately 
leaves S_IDLE and cop_busy — which is defined combinatorially as (state != S_IDLE) — goes high. 
cop_busy is wired directly to the stall input of the pipeline module. Inside pipeline.v, stall is regis tered into 
stall_read on the next clock edge, which holds fetch_pc constant (PC stops incrementing) and freezes 
both pipeline registers id_ex_reg and ex_mem_wb_reg so no new instructions commit. The pipeline stays 
frozen for the entire duration of the MAC computation — for a 784 -element dot product with 8 lanes this is 
approximately 784/8 = 98 accumulate cycles plus FSM overhead — before the stall releases and the 
CPU resumes exactly where it paused.  
Point 2 — LUT, FF, BRAM, and DSP usage estimates; will the design fit the target 
FPGA?  
Detailed estimates are provided in Section 4 above, targeting the Nexys A7 (Artix -7 XC7A100T). In 
summary: the design is estimated to consume approximately 20 of 135 available 36Kb BRAM tiles 
(RAMB36) (~15%), 72 of 240 DSP48E1 slices (30%) , 14,000 of 63,400 LUTs (~22%), and 10,000 of 
126,800 flip -flops (~8%) . All four figures are comfortably within device limits, leaving headroom for the 
systolic array goal system and stretch goals. The design will fit on the target FPGA. These estimates will 
be confirmed and refined after the first Vivado implementation run, e xpected by the Apr 12 milestone.
